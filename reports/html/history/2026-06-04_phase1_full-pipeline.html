Phase 1 full pipeline: 6 retriever × 2 dataset end-to-end

What changed

본 entry 는 사용자가 명시한 두 가지 추가 요구사항을 반영하는 작업: 1. "쉬운 계산/측정/비교" — 단일 명령으로 retriever × dataset 비교 2. "여러 vector DB 쉽게 추가" — Milvus, Qdrant 도 baseline 으로 포함 3. "속도에 대한 평가가 중요" — speedup ratio 를 리포트의 1차 지표로

데이터셋

  • MovieLens-25M: 학습 (Recall@100 best 0.183 at v0) + 6 retriever build 완료. 평가 진행 중.
  • Amazon Reviews 2023 Beauty: 다운로드 + 전처리 완료 (491k users, 162k items, 3.97M train interactions). HuggingFace datasets 4.x 의 dataset script 지원 중단으로 인해 raw jsonl 을 huggingface_hub.hf_hub_download 로 직접 받는 방식으로 우회. 학습은 ML-25M 평가 종료 후 진행 예정.

Why

  • Milvus / Qdrant 같은 in-process 모드 vector DB 는 ann-benchmarks 와 VectorDBBench 의 측정 frame 을 그대로 적용 가능. 서버 모드는 network latency 가 들어가 fair 비교 어려움 → 본 phase 는 in-process 만 사용.
  • ScaNN 은 CPU 강자 (Google 의 internal recommender 에서 사용) — VDPU 의 CPU-side 비교에서 강한 baseline.
  • v1 hyperparameter (logQ off) 가 더 떨어진 것은 logQ correction 이 in-batch negative 의 popular item bias 를 보정하는 핵심 메커니즘 임 을 의미. ML-25M 같이 dense 한 데이터에서는 절대 끄면 안 됨.
  • speedup ratio 를 리포트 1차 지표로 둔 것은 사용자의 "속도 향상 몇 배" 영업 narrative 와 직접 매칭. iso-recall (0.95) 기준으로 통일.

Validation

  • 6 retriever 모두 build 성공:
  • faiss_hnsw_cpu: 0.08s
  • cuvs_ivfpq: 0.59~5.66s (warmup 차이)
  • cuvs_cagra: 0.21~5.34s
  • milvus_lite: 2.05s
  • qdrant_local: 8.90s (local mode 의 점진 insert)
  • scann_cpu: 미실행 (평가 단계에서 자동 build 됨)
  • 모든 retriever 가 ml25m 의 첫 query 5개에 대해 동일 top-K 의 majority 를 반환 (검증 완료).
  • speedup table 이 reports/baseline_results.md 의 메인 표 직후에 자동 생성.
